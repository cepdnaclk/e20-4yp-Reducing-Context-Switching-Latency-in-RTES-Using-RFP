Key Definitions

DRFP

Dynamic Register File Partitioning: Heterogeneous logical partitioning of the physical register file.

CSR 0x800

Active descriptor managing partition base and size mapping for current context.

CSR 0x801

Staged descriptor written before trap return; committed by hardware on mret.

WCET

Worst-Case Execution Time: The critical metric for safety-critical timing determinism.

Oversubscription

Condition where runnable tasks exceed available resident hardware windows, triggering spill/reload fallback work.

Resident Path

Minimal switch path used when task context remains in a dedicated hardware partition.

O-WCET

Observed worst-case latency within the measured benchmark envelope, distinct from a certified universal bound.

Section 01 // Abstract

Eliminating the memory-bound bottleneck in context-switching.

"Context switching remains a major performance and predictability bottleneck in real-time embedded systems because saving and restoring general-purpose registers is memory-latency bound."

This project addresses a core real-time systems problem: context switches are often dominated by memory traffic from saving and restoring general-purpose registers. In a software-only path, that cost scales with the amount of architectural state moved to and from memory, which can hurt responsiveness and predictability as task pressure grows. Our approach, Dynamic Register File Partitioning (DRFP), is a hardware-software co-design built around CVA6, Spike, and FreeRTOS-Kernel. The objective is not to replace the operating system scheduler, but to reduce the expensive save/restore portion whenever a task can stay resident in a hardware-backed register partition.

On the hardware side, CVA6 is extended with a larger physical integer register file and two custom machine-mode CSRs: 0x800 for the active window descriptor and 0x801 for the staged/previous descriptor. The trap path snapshots and restores window metadata so that a resident task switch can avoid dense GPR spill/reload traffic. On the software side, the FreeRTOS RISC-V port is integrated with mixed context handling: a minimal path for window-resident tasks and a full save/restore fallback for non-resident or legacy paths. This keeps compatibility with the standard software model while enabling acceleration where the hardware residency condition is satisfied.

Spike is used as a functional oracle to validate CSR and trap semantics, while cycle claims are taken from CVA6 RTL simulation under Verilator. This separation is important: functional correctness and timing evidence are verified in different, complementary environments. We run dedicated bare-metal context-switch tests to isolate mechanism costs, then validate integrated behavior through FreeRTOS flow and regression scripts.

The measured results show a clear two-regime behavior. Under resident conditions, the handover primitive remains very small and stable, with low memory movement in the switch path. Under heavy oversubscription (for example, many logical tasks competing for a small number of resident windows), DRFP still preserves correctness through fallback, but average and tail latency increase because spill/reload work reappears. This makes the performance story realistic: DRFP reduces central switching cost when residency is available, and degrades gracefully when it is not.

In our published evaluation flow, this behavior is shown with stress and traffic/predictability benchmarks that compare DRFP against a software baseline in the same CVA6 environment. Representative numbers include lower average switch cost and significantly lower context-state bytes per switch on resident-friendly workloads, alongside wider spread in oversubscribed scenarios. These outcomes align with the architecture: the hardware primitive is short, but full end-to-end timing also depends on residency policy and fallback frequency.

The project therefore contributes a practical integration path rather than an isolated microarchitectural trick: custom CSR semantics in hardware, verified functional behavior in Spike, cycle-accurate evidence in CVA6 RTL, and OS-level application through FreeRTOS-Kernel. The software contract remains familiar (standard architectural register view for applications), while the platform gains a controllable mechanism to reduce context-switch overhead in real-time workloads.

Architectural Design

Hardware-level modifications to the RISC-V machine state for seamless partition management.

0x800

CSR_WINDOW_ACTIVE (0x800)

Active window descriptor. It selects the physical register partition currently visible as architectural x0-x31.

0x801

CSR_WINDOW_STAGED (0x801)

Staged/previous window descriptor. Software writes the next mapping here; trap return (\u006dret) commits it for the incoming task context.
